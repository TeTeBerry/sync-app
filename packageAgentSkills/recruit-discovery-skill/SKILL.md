# 公开招募检索

检索某场电音节的公开组队招募帖并展示列表卡片；仅信息检索，不配对撮合、不保证组满。

## 触发场景

用户原话举例（路由命中本技能）：
- 「帮我找去 EDC Korea 的公开组队帖」
- 「上海出发、3 人左右的 techno 招募」
- 「这场节有哪些人在找队」
- 「泰国 TML 组队」
- 「看看有没有一起去的」

## 不适用范围

- 智能匹配搭子、平台担保组满、私信配对 → 合规红线，不在本技能范围
- 自动发帖、代发招募 → 由草稿技能处理，须用户进小程序确认

## 前置条件

- 须已知具体活动（用户已选定或上文检索到节名）
- 公开招募列表无需登录；发帖草稿需登录

## 使用顺序

- 先确定活动，再按出发地、人数、曲风等条件检索公开招募帖
- 卡片点击可进入活动详情页查看完整招募墙

## 强制规则（MUST）

1. **卡片渲染强制规则**：调用 `searchPublicRecruits` 后**必须**让 `recruit-list-card` 卡片渲染生效，无论返回结果数量是多少（0 条、1 条、多条均须渲染）。**禁止**仅在文字中描述结果而不展示卡片。
   - 有结果时：卡片展示首条招募帖要点 + 剩余条数 + 筛选标签
   - 无结果时：卡片展示空状态，包含「暂无公开招募帖」文字 + 「写招募帖草稿」引导按钮

2. **活动名称强制规则**：所有文字回复和卡片内容中，**必须使用 `searchFestivals` 返回的 `events[].name` 完整官方名称**，**严禁**使用用户输入的简称或口语化名称。

3. **条件分支执行规则**：用户要求「先查 A 活动，没有则查 B 活动」时：
   - Step 1：先调用 `searchFestivals` 查 A 活动，拿到结果后再调用 `searchPublicRecruits` 查 A 的招募
   - Step 2：**仅在 A 活动 0 条招募帖时**，再调用 `searchFestivals` 查 B 活动 → 再查 B 的招募
   - **禁止**：跳过 A 的招募检索直接查 B；在 A 有结果时还去查 B

## 回复与展示要求

- 检索完成后**须展示招募列表卡片**（`recruit-list-card`），无论结果数量多少
- 有结果时：文字中概括首条招募帖要点，并说明还有多少条；卡片展示筛选标签
- 无结果时：明确告知「暂无公开招募帖」，卡片展示空状态并可引导用户起草招募帖
- 卡片点击打开活动详情招募墙（半屏）
- 活动名称必须使用完整官方名称，与 `searchFestivals` 返回一致
